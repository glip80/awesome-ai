Part 1 — Foundations

Core Prompt Types

Understanding prompt categories is the foundation of effective AI interaction. The main types work across all major models — GPT, Claude, and Gemini — and each serves a distinct interaction pattern.

TypePurposeExample
Zero-ShotQuick general response, no examples needed"Summarize this article in 5 bullets."
Few-ShotMimic a structure or tone using 2–3 examples"Here are 2 examples. Write a third in the same style."
InstructionalDirect commands with verbs: write, explain, compare"Write an exec summary under 100 words."
Role-BasedGive the AI a persona for domain-specific output"You are a senior Python engineer. Review this code."
ContextualAdd background framing before the question"This is for a fintech API docs audience."
ComparativeEvaluate or contrast multiple options"Compare Option A vs B in strengths and weaknesses."
Opinion-ElicitingSolicit the AI's viewpoint on a topic"What might happen if X regulation passes?"
Task-SpecificDrive a concrete action, used heavily in agents"Run this script and return only errors."

Advanced Patterns

Beyond basic types, patterns are reusable structures that solve recurring interaction problems. These are especially powerful in multi-turn conversations and agentic workflows.

Chain-of-Thought (CoT)

Add "Think step by step" before complex reasoning. Forces intermediate reasoning steps, dramatically improving accuracy on math, logic, and code.

Flipped Interaction

Ask the AI to question you until it has enough context: "Ask me questions one at a time until you can write a full project spec."

Tail Generation

Instruct the model to repeat key info at the end of each reply. Maintains context in long conversations where context window pressure builds.

Role Persona

Persistent role assignment in the system prompt: "You are a Python expert. Return only code, no explanation." Eliminates repeated context-setting.

RAG Prompting

Connect the model to your knowledge base so it answers with domain-specific accuracy. Chunk documents at ~512 tokens with overlap for best retrieval recall.

Output Constraints

Always define format, length, and audience in the prompt. Use few-shot examples in API calls when output format consistency matters (e.g., structured JSON).

Quality Techniques

These go beyond individual prompts into how you architect your overall AI usage for reliable, production-grade output.

Practical Tips for Engineers

  • Define role + task + constraints in every system prompt: "You are a Python expert. Rewrite the following function to be O(n). Return only code, no explanation."
  • Use few-shot examples in API calls when output format consistency matters — JSON responses, structured reports, typed code.
  • For RAG pipelines, chunk documents at ~512 tokens with overlap for better retrieval recall.
  • Use prompt caching (Anthropic/OpenAI) to cut evaluation costs by up to 90% when running 1000+ benchmark completions.
  • Test prompt variations with langchain or promptflow to benchmark which formulation yields the most reliable output.

Retrieval-Augmented Generation (RAG)

RAG is an architecture pattern that extends an LLM with access to external, up-to-date, or private knowledge at inference time — without retraining the model. Instead of relying solely on the model's training data, RAG retrieves relevant documents on demand and injects them into the prompt as grounded context.

Why RAG exists — the three LLM limitations it solves

1. Knowledge cutoff — LLMs know nothing after their training date. RAG adds a live knowledge base. 2. Hallucination — grounding the model on retrieved source documents dramatically reduces fabrication. 3. Private data — your internal docs, codebase, Jira tickets, and Confluence pages are never in any public model's training data. RAG makes them accessible.

The RAG Pipeline — 6 Stages

#StageWhat HappensKey Decisions
1IngestLoad raw sources: PDFs, web pages, Markdown, code, ticketsWhich sources? Update frequency?
2ChunkSplit documents into retrievable segmentsChunk size (~512 tokens), overlap (~50 tokens), strategy
3EmbedConvert each chunk to a dense vector via an embedding modelModel choice: text-embedding-3-small, all-MiniLM-L6-v2
4IndexStore vectors + metadata in a vector databaseDB choice: Qdrant, Chroma, Pinecone, pgvector
5RetrieveEmbed the query, similarity-search the index, return top-k chunksk value, similarity threshold, hybrid vs. dense search
6GenerateInject retrieved chunks into the prompt, LLM generates grounded answerContext window budget, citation instructions

Minimal RAG in Python (LangChain + Qdrant)

End-to-end RAG pipeline
from langchain_community.document_loaders import DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_qdrant import QdrantVectorStore
from langchain.chains import RetrievalQA

# ── Stage 1-2: Load & chunk ─────────────────────────────────────────
loader = DirectoryLoader("./docs", glob="**/*.md")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512, chunk_overlap=50
)
chunks = splitter.split_documents(docs)

# ── Stage 3-4: Embed & index ────────────────────────────────────────
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = QdrantVectorStore.from_documents(
    chunks, embeddings,
    url="http://localhost:6333",
    collection_name="my-knowledge-base"
)

# ── Stage 5-6: Retrieve & generate ──────────────────────────────────
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
llm = ChatOpenAI(model="gpt-4o-mini")

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True   # always verify your sources
)

result = qa_chain.invoke({"query": "How do I deploy to staging?"})
print(result["result"])
print("Sources:", [d.metadata["source"] for d in result["source_documents"]])

Chunking Strategies

Chunking is one of the highest-leverage decisions in a RAG pipeline — poor chunking directly degrades retrieval quality regardless of how good your embedding model or vector DB is.

Fixed-Size Chunking

Split every N tokens with an overlap window. Fast and predictable. Best for uniform documents (log files, structured records). Default: 512 tokens, 50-token overlap.

Recursive / Semantic

Split on paragraph → sentence → word boundaries in order. Preserves natural language boundaries. Best for prose documents, README files, and knowledge articles.

Document-Aware

Chunk by Markdown headers, HTML sections, or code functions. Keeps semantic units intact. Best for technical docs, API references, and code repositories.

Agentic / Late Chunking

Embed entire documents first, then chunk only the retrieved documents at query time. Slower on first call but cached results become faster. Preserves full document context.

Advanced RAG Patterns

PatternHow It WorksBest When
Hybrid SearchCombines dense vector similarity (semantic) with BM25 keyword scoring, merged via Reciprocal Rank Fusion (RRF)Domain with exact-match terms (product names, error codes, IDs)
Re-RankingInitial retrieval fetches top-20 candidates, a cross-encoder re-ranks to top-5 for the LLMHigh-precision tasks where top-1 accuracy matters
Query ExpansionLLM rephrases the query 2–3 ways before retrieval, results merged — improves recallAmbiguous or short user queries
Agentic RAGAgent decides when and how to retrieve — multi-hop, iterative, self-correcting retrieval loopsComplex multi-step questions requiring reasoning across sources
Self-RAGModel uses reflection tokens to decide if retrieval is needed at all, then evaluates relevance of what it retrievedMixed queries — some need retrieval, some don't
Multi-Modal RAGSeparate embedding pipelines for text and images (CLIP), unified retrievalDocument sets with diagrams, charts, screenshots

Common Failure Modes & Fixes

  • Chunks too large — dilutes signal in the embedding vector. Fix: reduce to 256–512 tokens, increase overlap to 10–15%.
  • Missing exact-match terms — vector search misses product names, error codes, IDs. Fix: add BM25 hybrid search alongside dense vectors.
  • Context stuffing — jamming all top-k chunks fills the context window with noise. Fix: re-rank before injection; use top-3, not top-10.
  • Stale index — knowledge base not refreshed when source docs update. Fix: incremental ingestion on document change events (webhooks or polling).
  • No source attribution — model answers without citing which chunks it used; impossible to verify. Fix: always return source_documents and render source links in your UI.

Part 2 — Agent Building Blocks

Agent Skills

Skills are modular, reusable capability packages — written once and loaded by any compatible agent. Each skill lives in a SKILL.md file (or similar) and is loaded automatically when the task description matches the skill's trigger keywords.

Best practices for writing skills

Keep each skill focused on one job. Write imperative steps with explicit inputs and outputs. Front-load trigger keywords so the agent can match it implicitly. Prefer plain instructions over scripts unless you need deterministic behavior.

Example SKILL.md structure
# Skill: generate-standup
# Trigger: "standup", "daily update", "git log summary"

description: "Generate a standup update from the git log of the last 24 hours."

steps:
  1. Run `git log --since="24 hours ago" --oneline`
  2. Group commits by ticket number or feature area
  3. Summarize in standup format: Done / In Progress / Blocked
  4. Output as plain text, max 5 bullets

Hooks

Hooks are lifecycle interceptors — they run automatically at specific points around agent execution. They fire before the agent's prompt loads, after completion, or around specific events like file edits.

HookWhen It FiresCommon Use Cases
before-workBefore the agent starts a taskValidate inputs, load context
after-workAfter the task completesLogging, notifications, CI triggers
before-editBefore a file is modifiedSecurity checks, backups
after-editAfter a file is modifiedLinting, formatting, tests
on-errorOn failureError reporting, Slack alerts
on-successOn successDeploy triggers, webhooks
on-retryOn retry attemptsBackoff logic, alternate strategies
Hook example — auto-lint after edit
# .hooks/after-edit.sh
if [[ "$EDITED_FILE" == *.py ]]; then
  ruff check "$EDITED_FILE" --fix
  black "$EDITED_FILE"
fi

Agent Prompt Architecture

When a prompt runs inside an agent, context assembles in a specific order. Understanding this stack lets you design more predictable, debuggable agents.

LayerWhat It ContainsEngineering Role
System PromptPersona, constraints, output formatSet once, persists every turn
SkillsLoaded SKILL.md files matching the taskReusable capability modules
Tool SchemasAPI/tool definitions the agent can callDefines agent's action space
HooksPre/post lifecycle instructionsValidation, logging, safety
Memory / ContextPrior session summaries, retrieved docsRAG, conversation history
User MessageThe actual task or queryWhat the agent acts on

Key Agent Prompting Patterns

  • Goal prompts over task prompts — tell the agent what success looks like, not just the next step
  • Memory continuity — "Before the next step, summarize: completed tasks, errors, tools used, decisions made"
  • Explicit tool invocation — "Use the search tool first, then the write tool, then confirm"
  • Chain-of-thought — ask agents to reason step-by-step to make debugging possible and reduce confident-but-wrong outputs

MCP — Model Context Protocol

MCP is an open standard that lets agents connect to external tools and data sources in a standardized way. Think of it as USB-C for AI agents — write one connector, use it across Claude, Codex, Cursor, and other compatible runtimes.

Context Engineering

Anthropic's term for managing the full context state: system instructions, tools, MCP connections, and message history. It extends beyond prompting into how you architect the entire information flow into the model.

Simple MCP server in Python
from mcp import Server, tool

server = Server("my-code-tools")

@tool
def run_tests(file_path: str) -> str:
    """Run pytest on a specific file and return the output."""
    result = subprocess.run(["pytest", file_path, "-v"], capture_output=True)
    return result.stdout.decode()

server.run()

Plugins

A plugin bundles multiple skills — and other components like tool connections — into a single installable, shareable package. If a skill is the recipe, a plugin is the cookbook. This distinction matters as your agent workflows grow in complexity.

Skill vs. Plugin — the core difference

Use a skill when you have one repeatable, self-contained workflow — deploying to staging, reviewing a PR, writing a status update. If the instructions fit in one file, a skill is enough. Use a plugin when you have a collection of related skills that should travel together, or when the workflow needs external tool connections (e.g., CRM, Jira, Slack).

DimensionSkillPlugin
ScopeOne specific, self-contained taskCollection of related skills + tool connections
File structureSingle SKILL.md fileDirectory with manifest, multiple skills, config
VersioningPer-file, manualPackaged and versioned as one unit
DistributionCopy/paste or Git submoduleInstall via registry or package manager
External toolsRarely neededBundles MCP connections (CRM, CI, APIs)
Example"Generate standup from git log""Sales enablement" — call prep + research + CRM sync

Prompts in Agent Context

In agentic applications, prompts have a strict hierarchy. Understanding each layer — and what belongs in each — is the foundation for building reliable, secure, and consistent AI systems.

Prompt TypeScopeSet ByChanges When
System PromptApplication-wide, all usersDeveloperProduct behavior changes
User PromptSession-specific, one interactionEnd userEvery turn
Agent PromptGoal-oriented, multi-stepDeveloper/orchestratorTask or tool context changes
Tool PromptSingle tool invocationTool schema definitionTool is called

System Prompts

System prompts establish application-wide behavior that persists across every user interaction in a session. They are ideal for defining brand voice, output formatting, safety constraints, and operational boundaries. The model gives system prompts the highest attention priority — they shape behavior even more than the user message.

Production system prompt pattern
# System Prompt Template
You are [role], a [description] assistant for [product/team].

Behavior rules:
- Respond only about [domain]. Politely decline out-of-scope questions.
- Always format code responses as fenced blocks with language tags.
- If the user asks you to do something dangerous, refuse and explain why.
- Default response language: English. Mirror user's language if they switch.

Output format:
- Use Markdown headings for structure.
- Keep answers under 400 words unless the user explicitly asks for more.
- Cite your reasoning when making non-obvious decisions.

User Prompts

User prompts are session-specific and transient — they contain the variable content that makes each interaction unique. They operate within the framework established by the system prompt, and can invoke specific roles or modes without permanently changing the agent's behavior for all future turns.

Position Bias — a production gotcha

Models prioritize messages based on their position. System messages receive the highest attention weight. In coding scenarios where original code is in the system prompt, the model may prioritize the initial code over revisions made later in the conversation. Keep mutable content in user messages, not the system prompt.

Agent Prompts vs. Standard Prompts

Agent prompts are far more detailed than standard LLM prompts — they serve as the primary control mechanism for complex, goal-oriented, and multi-step processes. Key differences:

Standard Prompt

One turn. User asks, model answers. No planning, no tool calls, no memory. Stateless by default.

Agent Prompt

Multi-step. Defines goal, available tools, fallback logic, memory instructions, and output contracts. Stateful across turns.

Resilience Instructions

Agent prompts include fallback logic: "If a search returns no results, rephrase with broader terms. If still no results, ask the user."

Tool Invocation Prompts

Explicitly define when and how to call tools: "Use the search tool first. Only write files after user confirms the plan."

Prompt Hierarchy Best Practices

  • System prompt = configuration layer — persona, safety constraints, output format, domain limits. Never put mutable or user-specific data here.
  • User prompt = request layer — the actual task. Can invoke a role ("As a financial analyst...") without permanently changing system behavior.
  • Inject RAG context in the user turn, not the system prompt — keeps the system prompt lean and the retrieved content close to the question.
  • Use the assistant role for multi-turn memory — include prior assistant responses in the message history to maintain reasoning continuity.
  • Keep system prompts under 800 tokens — bloated system prompts dilute attention on the core instructions and increase cost per call.

Part 3 — Benchmarking

PromptFlow (Microsoft)

PromptFlow models your pipeline as a Directed Acyclic Graph (DAG) where each node is an LLM call, Python function, or tool. This makes flows traceable and easy to debug visually — ideal for Azure-based teams and structured pipelines.

Benchmarking workflow
from promptflow.client import PFClient
import json

pf = PFClient()

# 1. Run base flow against a dataset
base_run = pf.run(
    flow="./my-flow",
    data="./data.jsonl"
)

# 2. Chain an eval flow onto the base run
eval_run = pf.run(
    flow="./evaluation/eval-classification-accuracy",
    run=base_run
)

# 3. Get metrics and open browser dashboard
metrics = pf.get_metrics(eval_run)
print(json.dumps(metrics, indent=4))
pf.visualize([base_run, eval_run])

Built-in PromptFlow Metrics

  • Accuracy, F1 score, GPT similarity
  • Relevance scoring, Ada similarity
  • Custom Python-based evaluators for domain-specific criteria

LangChain + LangSmith

LangChain's benchmarking power lives in LangSmith, its evaluation and observability platform. It is code-first and better for complex agent or multi-model comparisons. LangChain benchmarked 5 prompt optimization methods across 3 models, finding up to 200% performance improvement from automated prompt tuning.

Core LangSmith Workflow

  1. Create a dataset — a .jsonl of {input, expected_output} pairs in LangSmith UI or SDK
  2. Define an evaluator — LLM-as-a-Judge, exact match, or custom scoring function
  3. Run evaluation — run your chain/agent against the dataset
  4. Compare runs — diff prompt versions side-by-side with score graphs
LLM-as-a-Judge evaluator
from langsmith import Client
from langsmith.evaluation import evaluate

def correctness_evaluator(run, example):
    score = judge_llm.invoke(
        f"Question: {example.inputs['question']}\n"
        f"Expected: {example.outputs['answer']}\n"
        f"Got: {run.outputs['answer']}\n"
        f"Score 1-5:"
    )
    return {"score": int(score.content)}

evaluate(
    your_chain,
    data="my-benchmark-dataset",
    evaluators=[correctness_evaluator],
    experiment_prefix="prompt-v2"
)

PromptFlow vs. LangSmith

Both tools can be combined — LangChain inside a PromptFlow node gives you maximum flexibility. Choose based on your primary workflow and deployment environment.

5-Step Engineering Workflow

  1. Build a golden test set — 50–100 representative {input, expected} pairs covering edge cases and regressions
  2. Run on every prompt change — treat it like unit tests for your instructions
  3. Use LLM-as-a-Judge for open-ended outputs where exact match fails (code explanations, summaries)
  4. Track with LangSmith — tag experiments by model version, prompt version, and date
  5. Enable prompt caching to cut evaluation costs by up to 90% when running 1000+ benchmark completions
